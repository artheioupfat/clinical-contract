Documentation

clinical-contract validates YAML contracts and checks data files with a DuckDB-only engine. This page covers the practical model and execution flow.

What clinical-contract does

  • Validates required YAML structure before any file scan.
  • Checks schema compatibility between YAML logical types and dataset columns.
  • Executes quality SQL rules and reports pass/fail/error statuses.

Contract YAML structure

dataset: export
description: Export contract
qualityRules:
  - schema: export
    properties:
      - name: IPP
        logicalType: string
        quality:
          - id: ipp_not_null
            query: SELECT COUNT(*) FROM export WHERE IPP IS NULL
            expected: 0
      - name: DATE_EVENEMENT
        logicalType: timestamp

Type mapping policy

  • Strict matching for explicit integer widths (`int8..int64`, `uint8..uint64`).
  • Broad family matching for generic types (`integer`, `string`, `datetime`, `boolean`, `float`, `binary`).
  • Unsupported `logicalType` fails early during contract validation.

Validate then Check flow

  1. Validate: verifies mandatory YAML fields and shapes.
  2. Schema: compares expected logical types and detected dataset types.
  3. Quality: runs SQL checks and aggregates result statuses.

Data format support

The browser playground supports .parquet and .csv uploads. Preview uses page-by-page DuckDB queries (50 rows/page).

Browser limitations

  • PyScript/Pyodide has a cold start delay before runtime is ready.
  • Very large files increase in-browser memory and query latency.
  • Use CLI for heavy production-scale workloads and CI automation.
